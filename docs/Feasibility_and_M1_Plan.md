# DeskMochi 可行性与 M1 起步草案

Source baseline: `docs/DeskMochi_GDD_and_Roadmap.md`

## 当前判断

DeskMochi 的方向可行，但应该先把 M1 收敛成一个小而完整的桌面宠物原型：透明无边框窗口、可拖拽软体身体、基础状态机、低资源空闲策略。番茄钟、ToDo、Git/Token 监听这些功能都值得做，但不应该进入第一轮核心验证。

## 技术路线建议

### 引擎

优先使用 Godot 4.x。

原因：

- 透明窗口、无边框窗口、鼠标输入控制和跨平台打包都比较适合桌宠。
- 2D 渲染与物理足够轻量，适合常驻桌面。
- 项目可以用 GDScript 快速验证，后续性能瓶颈再局部迁移到 C# 或 GDExtension。

### M1 软体实现

建议先不追求真实复杂柔体。第一版采用“视觉柔体 + 简化物理”的折中方案：

- 用一个中心刚体或位置状态表示整体身体。
- 用若干控制点或虚拟骨架点表示软体轮廓。
- 鼠标拖拽/戳按产生局部形变参数。
- 渲染层根据形变参数绘制或驱动精灵/网格变形。

这样能快速做出“糯叽叽”的手感，同时避免一开始陷入复杂稳定性调参。

### 架构

保持单向数据流：

- Input 层采集鼠标、窗口、系统事件。
- Simulation 层更新宠物状态、位置、形变、计时器。
- Presentation 层只读取状态并渲染，不承载业务逻辑。
- Effects 层把状态变化映射成动画、粒子、音效等表现。

避免 Godot Autoload 单例作为业务核心。必要的根节点可以持有状态树，并把依赖显式传给子模块。

## M1 验收标准

- 应用启动后显示一个透明背景、无边框、置顶的桌面窗口。
- 宠物主体可以被鼠标拖拽移动。
- 点击、戳按或快速拖动时有明显局部压缩、拉伸和回弹。
- 支持基础状态：Idle、Dragged、Poked、Falling、Settled。
- 空闲状态下 CPU/GPU 占用保持低水平。
- 代码结构能清晰分辨输入、状态更新和渲染表现。

## 建议的第一周任务

1. 初始化 Godot 4 项目结构。
2. 实现透明无边框窗口和置顶显示。
3. 建立最小单向数据流框架。
4. 做一个临时圆形/椭圆形麻糬主体。
5. 实现拖拽、戳按、回弹三种核心交互。
6. 写一份短的 M1 技术决策记录，记录哪些能力暂缓。

## 暂缓项

- 自定义五官和头饰。
- 番茄钟、ToDo、闹钟、播放器。
- 键盘监听、Git 监听、Token 监听。
- 复杂粒子庆祝动画。
- 分发平台 SDK。

这些功能依赖核心桌宠体验成立之后再接入。

## 需要确认的问题

1. 第一版目标平台是否只做 Windows，还是从一开始就要求 Windows/macOS/Linux？
2. M1 更看重“真实物理”还是“软萌手感”？我建议优先手感。
3. Godot 脚本语言是否接受先用 GDScript？后续需要时再引入 C#。
4. 视觉资产第一版是否可以先用程序绘制/占位图，等交互手感确认后再换正式美术？
